# 版本记录：v1.11-cs-foundation-learning-map

## 1. 本轮目标

本轮目标是按用户明确给出的学习规划，重写首页学习进度地图，使路线从基础能力开始展开，而不是继续展示偏终局成就的科研/金融技能树。

本轮目标类型属于功能内容更新与可维护性提升：核心是更新 `learningMapData` 的节点、依赖、状态和描述，同时修复地图组件中已经存在的中文乱码文案，使用户点击节点时能看到可读的状态与说明。

## 2. 背景与上下文

本轮承接以下历史记录：

- `custom/notes/v1.8-learning-ui-redesign.md`：确认学习地图应作为可拖拽、可缩放、可点击的课程依赖图，而不是普通卡片列表。
- `custom/notes/v1.10-minecraft-learning-map-and-cc98-fix.md`：确认地图数据集中在 `src/data/learningMap.ts`，组件入口集中在 `src/components/LearningMap.tsx`。
- `custom/notes/v1.10-apple-ui-redesign-and-404-fix.md`：确认后续应保留 Apple/Glassmorphism 视觉基础，不再做大范围 UI 换皮。

修改前，地图中包含金融、科研、深度学习、控制、机器人项目等偏高级或成果导向的节点，并且部分状态被标成已完成。用户这次明确要求改为“从基础开始”，并指定基础包括数学基础、编程基础、算法基础、深度学习和机器人学。

## 3. 问题分析

原地图的核心问题不是交互能力不足，而是内容结构与当前学习阶段不匹配：

- 数学基础过于粗略，只包含微积分、线性代数和凸优化，缺少概率论与数理统计这一深度学习和强化学习前置基础。
- 编程基础原本把 Python、C、C++ 标成偏完成状态，但用户当前明确提出这些应作为需要推进的基础项。
- 深度学习、机器人学节点出现过早，如果没有数学、编程、算法和系统基础支撑，地图容易给出错误优先级。
- `LearningMap.tsx` 中存在一批中文乱码，会影响节点弹窗、状态标签和按钮的可读性。

因此本轮的核心矛盾是：既要保留已建立的互动地图框架，又要把内容重排为更真实的学习路线。

## 4. 候选方案比较

方案 A：保留旧地图，只在后面追加用户指定节点。

- 优点：改动最少，视觉布局几乎不变。
- 缺点：旧地图中的高级节点和完成状态会继续误导学习优先级，新旧路线混杂。
- 风险：后续继续迭代时难以判断哪些节点代表真实进度，哪些只是展示性内容。
- 结论：放弃。

方案 B：完全重写地图数据，但保留 React Flow 组件和视觉系统。

- 优点：内容结构能完全贴合用户当前规划，改动范围仍限制在地图数据和少量文案。
- 缺点：需要重新手写节点坐标和依赖关系，短期内布局仍依赖人工维护。
- 风险：如果用户实际完成度与默认状态不同，后续还需要继续微调 status 字段。
- 结论：采用。

方案 C：引入自动布局或更复杂的课程分类字段。

- 优点：长期可扩展性更好。
- 缺点：会引入新的实现复杂度，超过本轮“按用户规划写进度”的范围。
- 结论：暂不采用。

## 5. 最终决策

采用方案 B：只重写 `src/data/learningMap.ts` 的学习节点数据，并修复 `src/components/LearningMap.tsx` 中的中文乱码与重复 `className` 属性。

本轮路线按用户规划拆成五段：

1. 数学基础：微积分、3Blue1Brown 线性代数、概率论与数理统计、Stanford EE364A 凸优化。
2. 编程基础：CS61A Python 程序抽象、C 语言基础、CS106L 现代 C++。
3. 算法与系统基础：CS61B 数据结构基础、CSAPP 计算机系统。
4. 深度学习：CS231n 计算机视觉、CS224N 深度学习与 NLP、强化学习。
5. 机器人学：作为最终汇合节点，依赖 C++、CSAPP、凸优化和强化学习。

## 6. 具体实现

修改文件如下：

- `src/data/learningMap.ts`：删除旧的金融/科研成就树节点，替换为用户指定的基础学习路线。节点状态上，微积分、3Blue1Brown 线性代数和 CS61A 标为 `in-progress`，C 语言基础标为 `available`，其余高阶节点按依赖标为 `locked`。
- `src/components/LearningMap.tsx`：修复悬浮提示、地图标题、弹窗状态、任务日志、经验值、前置要求、按钮文案中的中文乱码；移除重复 `className` 属性；保留 React Flow、Framer Motion、节点主题和点击弹窗结构。
- `docs/`：执行 `npm run build` 后自动更新静态站点产物和 hashed asset 文件。

## 7. 本轮优点

- 学习路线回到用户当前真正需要推进的基础能力，而不是展示性成就树。
- 依赖关系更清晰：数学和编程基础先行，算法/系统承接，深度学习和机器人学后置。
- 地图组件的中文可读性明显改善，弹窗不再出现乱码。
- 没有引入新依赖，没有改动全站视觉架构，风险集中在学习地图模块内。

## 8. 本轮缺点与代价

当前方案仍有以下代价：

- 节点坐标仍是手写，后续节点继续增多时可能需要重新排版。
- status 字段基于用户“从基础开始”的描述进行保守设定，并不代表真实完成度的精确记录。
- CS61B 使用 Java 生态，但用户将其列为数据结构基础，因此本轮保留课程名而不额外引入替代课程。
- CS224N 更准确地说是深度学习与 NLP 课程，本轮按用户表述放在“深度学习”分支中，后续可视需要再拆分为“深度学习基础”和“NLP”两级。

## 9. 验证与结果

已执行验证：

- `npm run build`：成功。Astro 构建出 8 个静态页面，并更新 `docs/` 产物。

验证限制：

- `npm run check`：失败，但失败点集中在仓库根目录已有的历史临时脚本 `scaffold_new.cjs`、`script2.js`、`update-homepage.js`，报错包括非法字符、未闭合字符串、重复变量声明等；这些文件不是本轮修改范围。本轮没有擅自删除或修复这些临时脚本。

## 10. 本轮结论

本轮确认学习地图最适合继续作为“依赖图”维护，而不是普通进度列表。对于当前阶段，正确优先级应是：

先数学基础和编程基础，再数据结构与系统，再深度学习，最后汇合到机器人学。

本轮也确认地图组件自身没有必要大改，当前主要问题来自数据和中文文案。

## 11. 下一轮建议

1. 根据用户真实完成情况微调每个节点的 `status`，尤其是微积分、线性代数、CS61A、C 语言基础这几个起点。
2. 如果后续节点继续增加，再考虑引入自动布局算法，避免手写坐标越来越难维护。
3. 单独处理根目录历史临时脚本，避免 `npm run check` 被无关脚本长期阻断。
