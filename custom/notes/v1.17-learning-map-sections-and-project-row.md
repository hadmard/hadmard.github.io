# 版本记录：v1.17-learning-map-sections-and-project-row

## 1. 本轮目标

按用户最新要求，将学习地图的连线逻辑从“多交叉网状”改为“分成几个部分”，并在末端“汇总成项目行”，降低视觉混乱。

## 2. 背景与上下文

本轮先读取：

- `AGENTS.md`
- `custom/notes/v1.16-learning-map-mc-style-static.md`
- `custom/notes/v1.15-learning-map-visual-refresh.md`
- `custom/notes/v1.11-cs-foundation-learning-map.md`
- `src/data/learningMap.ts`
- `src/components/LearningMap.tsx`
- `src/components/HomePage.astro`

延续思路：

1. 保留 v1.16 的 MC 风格静态交互（不可拖动，保留悬停与点击详情）。
2. 保留“学习路线可视化”的主目标。

放弃思路：

1. 放弃当前大量跨分区依赖导致的密集连线。
2. 放弃“每个节点都与多个前置直接连线”的展示方式。

## 3. 问题分析

用户反馈“线条太乱”，核心原因是：

1. 同一节点常有多个跨列前置，边线交叉密集。
2. 分区边界不清晰，阅读路径不够一眼可见。
3. 缺少“终点汇总层”，学习结果难以聚焦到项目产出。

## 4. 候选方案比较

方案 A：仅调整边样式（颜色/粗细/透明度），不改依赖。  
优点：改动最小。  
缺点：交叉关系仍在，乱线问题本质不变。

方案 B：重排依赖结构为“分区内主链 + 少量跨区桥接 + 项目行汇总”。  
优点：从结构层面减少线条，阅读路径更清晰。  
缺点：需要调整节点依赖与坐标。

最终采用方案 B。

## 5. 最终决策

1. 将数学、编程、系统、AI 统一改为各自纵向主链。
2. 仅保留必要跨区桥接依赖，减少横向交叉线。
3. 新增“项目行”三节点，作为最终汇总层。
4. 保留机器人学节点作为能力汇合，再接项目行终点。

## 6. 具体实现

### 6.1 数据层重排

文件：`src/data/learningMap.ts`

1. 数学线改为串行依赖：微积分 -> 线代 -> 概统 -> 凸优化。
2. 编程线改为串行依赖：CS61A -> C 基础 -> C++。
3. 系统线改为串行依赖：CS61B -> CSAPP。
4. AI 线改为串行依赖：CS231n -> CS224N -> 强化学习，并保留与数学优化的必要桥接。
5. 机器人学节点依赖精简为系统与强化学习主桥接。
6. 新增项目行三节点：
   - 项目A 机器学习系统实战
   - 项目B 系统与性能工程
   - 项目C 机器人综合集成（汇总终点）
7. 统一重排坐标，形成“分区列 + 底部项目行”的结构。

文件：`src/components/HomePage.astro`

- 地图区说明改为“split into clear sections, ending with a unified project row”。

### 6.3 流程文件同步

文件：`AGENTS.md`

## 7. 本轮优点

1. 从依赖结构层面减少乱线，不只是样式层面的掩盖。
2. 分区更清楚，阅读路径更接近“先学基础，再做项目”。
3. 项目行强化“学习成果导向”，更符合用户表达。

## 8. 本轮缺点与代价

1. 依赖被简化后，部分真实多前置关系没有全部显式展示。
2. 若后续节点继续增加，仍可能需要自动布局或分屏视图。

## 9. 验证与结果

本轮执行最小可行验证：

- `npm run build`

验证预期：

1. 构建成功。
2. 首页地图可正常渲染。
3. 交互仍满足不可拖动 + 悬停 + 点击详情。

## 10. 本轮结论

本轮已将学习地图由“网状乱线”切换为“分区主线 + 项目行汇总”的结构，符合用户“分几个部分、最后汇总成项目行”的要求。

## 11. 下一轮建议

1. 增加“仅看主线/显示全部依赖”开关，让用户可在简洁与完整之间切换。
2. 为项目行增加真实里程碑（代码仓库、报告、Demo）字段，强化可执行性。
3. 如需进一步降低视觉负担，可把分区标题做成固定分栏标签。
