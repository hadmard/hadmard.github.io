# 版本记录：v1.6-cc98-authenticated-crawl

## 1. 本轮目标

本轮目标是把上一轮的“支持 token 的 CC98 API 爬虫”继续推进为“支持用户名密码自动登录并完成真实抓取”的版本。

这一轮属于：功能补充 + 真实链路打通。

优先级高的原因是，用户已经明确授权我使用其 CC98 登录信息继续抓取，那么当前最有价值的动作就不是继续手工整理，而是把真实登录和真实抓取流程真正跑通。

## 2. 背景与上下文

本轮承接：

- `custom/notes/v1.5-cc98-manual-record-import.md`
- `custom/notes/v1.4-cc98-investment-crawler.md`

上一轮已经形成两层成果：

1. 工具层：已有支持 token 的 CC98 API 爬虫。
2. 交付层：基于用户原文手工整理出一份本地投资记录。

但上一轮仍然存在一个明显限制：

- 想要真实抓取，仍然需要用户自己先提供 token。

本轮用户进一步直接提供了可用于登录的账号信息，因此当前重点转为：

- 不再停留在 token 级输入
- 直接把账号密码登录流程接入爬虫

## 3. 问题分析

表面上看，这一轮像是“加一个登录功能”。  
真正的核心问题是：

- 到底应该走浏览器模拟登录，还是直接走 CC98 的 OpenID 登录接口

我先做了验证，发现 CC98 前端当前并不是必须依赖浏览器表单提交，而是直接走：

- `https://openid.cc98.org/connect/token`
- `grant_type=password`

并且用网页端当前的 `client_id / client_secret` 换取 `access_token`。

因此当前关键矛盾不在于“如何模拟浏览器操作”，而在于“是否应该把已验证有效的 OpenID 登录流程直接接进脚本”。

## 4. 候选方案比较

### 方案 A：继续用浏览器自动化模拟登录

核心思路：

- 打开 `/logon`
- 自动填用户名和密码
- 从浏览器本地存储里取 token

优点：

- 更贴近用户真实网页行为
- 对网页改动的感知更直观

缺点：

- 额外依赖浏览器自动化
- 登录完成后还要再从浏览器状态提取 token
- 比直接走 OpenID 链路更重

放弃原因：

不是当前最短路径。

### 方案 B：直接把 OpenID 密码登录流程接入脚本

核心思路：

- 在脚本里直接请求 `connect/token`
- 用用户名密码换取 `access_token`
- 再按原有 API 抓取流程继续跑

优点：

- 路径最短
- 不引入额外依赖
- 和现有 token 抓取逻辑天然兼容

缺点：

- 依赖 CC98 当前网页端的登录实现细节
- 如果上游修改 `client_id / client_secret` 或 grant 方式，需要同步调整

最终选择原因：

这是当前成本最低、稳定性足够、与现有脚本最一致的方案。

## 5. 最终决策

本轮最终采用方案 B，具体决策如下：

1. 为爬虫新增 `--username` / `--password` 参数。
2. 同时支持环境变量：
   - `CC98_USERNAME`
   - `CC98_PASSWORD`
3. 保留原本的 token 直传模式，登录方式按以下优先级：
   - 直接 token
   - 用户名密码自动登录
4. 用真实登录跑通目标主题，生成一份授权抓取版输出。

## 7. 本轮优点

本轮的主要优点：

1. 真正打通了从登录到抓取的完整链路。
2. 不再要求用户先自己提取 token。
3. 继续沿用上一轮脚本主体，改动集中、边界清晰。
4. 可以直接产出授权抓取版记录，便于和手工导入版对照。

## 8. 本轮缺点与代价

这一轮也有明确代价：

1. 当前登录能力依赖 CC98 网页端现有 OpenID 密码登录实现。
2. 如果上游改了登录流程，脚本需要跟着调整。
3. 因为涉及真实账号登录，后续使用时必须更加注意不要把凭据写入仓库或日志。

## 9. 验证与结果

本轮完成的验证：

- 从前端包中确认登录流为 `grant_type=password`
- 用真实账号成功换取 access token
- 用该 token 访问：
  - `https://api.cc98.org/Topic/6450962/post?from=0&size=20`
- 成功拿到 10 条楼层数据

当前确认结论：

- 账号登录链路可用
- 目标主题可通过授权 API 正常抓取

## 10. 本轮结论

本轮最重要的结论是：

对于当前 CC98，真实抓取并不需要浏览器自动化；直接对接其 OpenID 密码登录流程就足够把授权抓取打通。

## 11. 下一轮建议

下一步最值得做的事：

1. 继续用新登录能力重新抓一遍目标主题，生成授权抓取版输出。
2. 如果你后续还要抓更多主题，可以直接复用用户名密码模式批量导出。
3. 之后再考虑把手工导入版与授权抓取版做一次差异校验。
