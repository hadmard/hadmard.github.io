# 版本记录：v1.7-cc98-pages-sync

## 1. 本轮目标

本轮目标是把已经通过 CC98 API 抓下来的主题数据正式同步到站点首页，并满足三个明确要求：

1. 支持按分页抓取全部楼层，而不是只停留在第一页。
2. 把每层楼的发帖时间完整写入本地文件并在页面上展示。
3. 在 GitHub Pages 页面上明确补充“来自 cc98.org”的来源标识。

这一轮属于功能补充 + 交付闭环。优先级高的原因是，上一轮虽然已经打通了真实抓取，但本地数据和站点页面之间仍然断开，用户实际感受到的成果还不完整。

## 2. 背景与上下文

本轮承接：

- `custom/notes/v1.6-cc98-authenticated-crawl.md`
- `custom/notes/v1.5-cc98-manual-record-import.md`

在开始本轮之前，仓库已有两层能力：

1. 爬虫层：`crawl-cc98-topic.mjs` 已经能够用登录授权抓取 CC98 主题，并按分页拉取楼层数据。
2. 文件层：`custom/cc98-investment-crawler/output/topic-6450962/` 下已经有 `raw/topic.json`、`raw/posts.json`、`records/investment-summary.json` 等本地文件。

但首页仍然使用站点内置的静态投资文案，没有真正消费这批本地同步数据，也没有把“来源于 cc98.org”这件事展示出来。

## 3. 问题分析

表面问题看起来只是“把数据渲染到首页”。

真正的核心矛盾是：

1. 数据已经存在，但 GitHub Pages 构建流程没有把它纳入首页渲染。
2. 用户关心的是“原文、楼层、时间”三类信息的保真同步，而不是再做一层摘要改写。
3. 如果首页继续依赖硬编码文案，就会造成“本地有真实记录，站点却看不到”的交付断层。

另外还有一个小但重要的细节：来源链接不能只写死在页面里，更合理的做法是直接读取本地摘要文件中的 `sourceUrl`，让站点展示和本地记录保持同源。

## 4. 候选方案比较

### 方案 A：继续只保留本地文件，不接入站点渲染

核心思路：

- 维持爬虫输出不变。
- 不改首页组件。

优点：

- 改动最小。
- 不会碰到前端构建逻辑。

缺点：

- 不能满足“同步到 GitHub Pages”的明确需求。
- 用户仍然需要自己进文件夹查看记录。

放弃原因：

这不构成交付闭环，等于只完成了一半。

### 方案 B：在首页直接硬编码帖子内容

核心思路：

- 把楼层文本手动写进 `src/data` 或组件里。

优点：

- 页面能很快显示内容。

缺点：

- 和本地抓取结果脱节。
- 每次帖子更新都要手改前端代码。
- 不符合“本地文件再同步到 GitHub Pages”的数据流。

放弃原因：

这会把同步关系做反，后续维护成本高。

### 方案 C：首页构建时直接读取本地抓取产物

核心思路：

- 在 `HomePage.astro` 构建阶段读取 `custom/cc98-investment-crawler/output/topic-6450962/` 下的 JSON。
- 直接展示原文、楼层、发帖时间、帖子 ID 与来源信息。

优点：

- 数据源和页面展示保持一致。
- 不需要额外服务端接口。
- 继续沿用现有 Astro 静态构建流程，改动边界清楚。

缺点：

- 首页组件需要承担一次本地文件读取逻辑。
- 如果将来支持多个主题，还需要再抽象同步入口。

最终选择原因：

这是当前最符合“最小改动 + 可验证同步 + 保留原文”的方案。

## 5. 最终决策

本轮最终采用方案 C，并做了两个关键决策：

1. 让首页组件在构建时直接读取本地同步文件，而不是重新请求 CC98 API。
2. 来源链接优先读取 `records/investment-summary.json` 的 `sourceUrl`，避免页面再写死来源地址。

这意味着数据流变成：

CC98 抓取结果 -> 本地 JSON 文件 -> Astro 构建 -> GitHub Pages 页面。

## 6. 具体实现

本轮实际改动如下。

### `src/components/HomePage.astro`

作用：

- 新增本地 JSON 读取逻辑。
- 从 `custom/cc98-investment-crawler/output/topic-6450962/` 读取主题和楼层数据。
- 在投资记录区域优先渲染同步后的 CC98 楼层内容。
- 展示楼层号、发帖时间、帖子 ID、作者和原文。
- 在页面上增加“来自 cc98.org”来源标识和原帖链接。
- 去掉调试期临时加入的 `console.info` 输出，保持正式构建输出干净。

### `custom/notes/v1.7-cc98-pages-sync.md`

作用：

- 固化本轮“本地文件同步到 GitHub Pages”的实现背景、取舍和边界。

### `AGENTS.md`

作用：

- 更新最新一期正式记录路径，保证后续迭代能先读到本轮结论。

## 7. 本轮优点

本轮带来的主要正向效果：

1. 真正把抓取结果接入了站点，不再只是目录里的离线文件。
2. 首页现在展示的是原帖楼层原文，保留了时间和楼层序号，符合“按原文就行”的要求。
3. “来自 cc98.org”的来源标识已经进入 GitHub Pages 页面，来源边界更清晰。
4. 来源链接来自本地摘要文件，减少了页面里的硬编码依赖。

## 8. 本轮缺点与代价

本轮仍有明确代价和未完美之处：

1. 当前首页同步入口仍然是针对 `topic-6450962` 的单主题实现，尚未抽象成多主题配置。
2. 首页组件承担了本地文件读取职责，后续如果同步主题变多，组件会继续膨胀。
3. 页面展示的是原文保真内容，没有进一步拆分成结构化交易日志卡片；这次是有意不扩展范围。

## 9. 验证与结果

本轮执行了最小可行验证：

1. 运行 `npm run build`
2. 检查构建产物中的首页 HTML 是否包含同步后的投资记录内容

验证结果：

- 构建成功。
- 生成的 `docs/index.html` 已经包含：
  - “投资记录已同步到站点首页”
  - “来自 cc98.org”
  - 原帖链接
  - 10 层楼的原文与发帖时间

未执行的验证：

- `npm run check`

原因：

- 之前相关检查在 Astro 诊断阶段触发过 Node 内存不足，本轮优先保留最直接、最相关的构建验证。

## 10. 本轮结论

本轮最重要的结论是：

对于当前仓库，最合适的同步方式不是再做一个在线接口，而是直接把本地抓取产物接入 Astro 构建阶段。这样既满足了 GitHub Pages 的静态站点约束，也保留了原帖内容的可追踪来源。

被进一步确认的判断：

1. 分页抓取的本地 JSON 结构已经足够支撑页面展示，不需要再二次抓取。
2. “原文 + 楼层 + 时间 + 来源”是当前这条投资记录最重要的四个展示维度。

## 11. 下一轮建议

下一轮最值得做的 1~3 件事：

1. 把首页同步入口整理成可配置主题列表，这是后续扩展多条 CC98 投资记录的最高收益方向。
2. 在不破坏原文展示的前提下，追加一层交易日志视图，把方向、标的、止损、复盘拆出来。
3. 如果后续帖子继续更新，直接重新运行抓取脚本并重新构建站点即可完成增量同步，风险最低。
