# 版本记录：v1.2-pages-branch-deploy

## 1. 本轮目标

本轮目标是把当前站点的 GitHub Pages 部署方式改成用户指定的“按 GitHub 官方快速入门，从分支发布”。

具体目标：

- 不再依赖 GitHub Actions 构建发布
- 改为由 `main` 分支的 `docs/` 目录直接作为 Pages 发布源
- 让仓库结构和 GitHub 文档中的配置方式一致

这一轮属于：部署策略调整 + 工程交付方式修正。

## 2. 背景与上下文

本轮承接：

- `custom/notes/v1.1-profile-seeding.md`

上一轮已经完成了站点基线与第一批真实资料接入，但部署方式仍是：

- GitHub Actions 构建
- `dist/` 产物上传
- Pages 走 Actions 发布

用户随后明确贴出了 GitHub Pages 官方快速入门文档，并要求“按照这个来”。

这意味着当前目标不再是“自动化程度最高”，而是“部署方式要和 GitHub 官方的分支发布心智模型一致”。

## 3. 问题分析

当前的核心问题不是站点本身，而是部署方式与用户认知不一致：

1. 对 Astro 来说，Actions 部署是更现代的做法，但它和 GitHub 文档里的“从分支部署”路径不一致。
2. 如果继续保留 Actions，用户之后在仓库 `Pages` 设置页里看到的流程会和文档说明对不上。
3. GitHub 官方分支发布适合简单、直观、可见的静态产物，但代价是需要把构建输出一并提交到仓库。

因此本轮的关键取舍是：

- 自动化优先
- 还是部署心智一致性优先

在用户已经明确指定的情况下，后者优先级更高。

## 4. 候选方案比较

### 方案 A：保留 GitHub Actions 部署

优点：

- 更自动化
- 不需要提交构建产物
- 更符合 Astro 官方推荐流程

缺点：

- 和用户贴出的 GitHub Pages 快速入门不一致
- 用户在仓库设置里会看到不同心智模型

放弃原因：

不符合用户当前明确要求。

### 方案 B：改为 `main` + `/docs` 分支发布

优点：

- 完全符合 GitHub 文档中的“从分支部署”
- 站点源码和构建结果都在仓库中可见
- 后续用户只需要记住“构建后提交 docs”

缺点：

- 仓库会包含构建产物
- 每次内容更新都需要重新生成 `docs`

最终选择原因：

与用户当前要求最一致，且实现成本低、行为可预期。

## 5. 最终决策

本轮最终采用：

- 删除 `.github/workflows/deploy.yml`
- 在 `astro.config.mjs` 中把构建输出目录改为 `docs/`
- 增加 `.nojekyll`
- 更新 `README.md`，明确说明 GitHub Pages 设置方式为：
  - `Deploy from a branch`
  - `main`
  - `/docs`

## 6. 具体实现

本轮主要改动文件如下：

- `astro.config.mjs`
  新增 `outDir: './docs'`
- `.github/workflows/deploy.yml`
  删除原先的 Actions 部署工作流
- `public/.nojekyll`
  确保 GitHub Pages 不走 Jekyll 处理
- `README.md`
  改成分支部署说明
- `AGENTS.md`
  更新最新一期记录路径

## 7. 本轮优点

当前方案的主要优点：

- 与 GitHub Pages 官方快速入门保持一致
- 用户后续理解和排查部署问题会更直观
- 仓库即发布源，交付路径更加可见

## 8. 本轮缺点与代价

本轮的 trade-off 很明确：

1. 需要把构建产物提交到仓库
2. 每次改动站点后，都必须重新运行 `npm run build`
3. 相比 Actions 方案，自动化程度降低

这些代价是主动接受的，因为用户明确要求优先遵循 GitHub 文档中的分支部署方式。

## 9. 验证与结果

本轮计划验证：

- `npm run check`
- `npm run build`

目标验证点：

- 构建输出进入 `docs/`
- 站点仍能正常生成
- GitHub Pages 可从 `/docs` 目录读取静态站点

## 10. 本轮结论

本轮最重要的结论是：

当用户明确指定 GitHub Pages 的发布方式时，部署心智一致性比更“现代”的自动化方案更重要。

## 11. 下一轮建议

下一步最值得做的事：

1. 在 GitHub 仓库设置里确认 Pages 的源切到 `main /docs`
2. 刷新线上站点确认新版本是否已正确生效
3. 继续补真实项目与文章内容
