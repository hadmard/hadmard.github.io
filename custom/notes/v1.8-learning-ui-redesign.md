# 版本记录：v1.8-learning-ui-redesign

## 1. 本轮目标

对系统整体 UI 体验进行全局重构，将传统的后台列表展现形式，转换为强调“探索感、节点式成长”的学习型产品界面。实现一个带有前后置依赖关系、可缩放拖拽的“课程进度图谱”（类似科技树 / 技能树）。

## 2. 背景与上下文

本轮承接：
- `custom/notes/v1.7-cc98-pages-sync.md`（上一轮同步了来自 CC98 的真实记录数据）

在此前阶段，系统的首页面板与主题数据展示虽然已经走通链路，但视觉展现停留在比较基础的卡片平铺和白底灰块风格，缺乏系统性与层级感。用户希望产品不仅是信息的容器，更是有“成长反馈和推进感”的数字工具。

## 3. 问题分析

现有的信息架构将课程/信息作为扁平列表列出，这在表达“学习路线”时会有诸多局限：
1. 无法直观展示“由于 A 未完成，所以 B 未解锁”的依赖关系。
2. 缺乏主次干线的区分。
3. 纯静态卡片缺乏“升级、连通”的动效正向反馈。

## 4. 候选方案比较

关于课程进度核心视图的实现：

**方案 A：纯原生 HTML + DOM + CSS 绝对定位连接**
- 优点：零额外依赖。
- 缺点：支持拖拽、平移缩放、自适应节点计算的成本极高，容易出现性能问题和边界 bug。

**方案 B：引入 D3.js**
- 优点：强大无所不能的数据可视化。
- 缺点：学习与编写心智偏重，对简单的节点连线图而言过于底层，不易与现代组件系统融合。

**方案 C：引入 React + XYFlow (`@xyflow/react`)**
- 优点：节点即组件，可直接复用 Tailwind 样式；内置缩放、连线、小地图、拖拽手势；生态成熟。
- 缺点：由于当前是 Astro 项目，需额外开启 React 集成。

## 5. 最终决策

根据目标，采取**方案 C**，在 Astro 中桥接 React 生态处理这种高度交互式的区域。
同时，根据业务特性敲定如下共识：
- 根据反馈最终决定采用保留明亮质感的浅色模式，保持更适合大篇幅阅读的纯净与克制。
- 重构数据结构字段，将扁平信息拓展为含有 `id`, `dependsOn`, `status` (`locked` | `available` | `in-progress` | `completed`) 等属性的有向无环图结构。

1. **浅色模式基线**：重写了 `global.css` 与 `Layout.astro`，恢复并强化了浅态下的微距阴影（Lift Shadow）和玻璃拟态。
2. **图谱核心数据**：抽象出 `learningMapData`（节点状态、XY位置、前后置依赖数组）。
3. **图谱组件实装**：开发了 React 组件 `LearningMap.tsx` 并将其外挂到 `HomePage.astro` 中：
   - 定制了四个状态（未解锁、可开始、进行中、已完成）的自定义渲染节点。
   - `onNodeClick` 挂载了详情侧滑抽屉，点击节点后弹出具体描述与任务开关。
4. **全动态连线**：图谱连接线会自动根据前置依赖进行发光流动（Animation Dashoffset）。

## 7. 本轮优点

1. 产品从传统的瀑布流/博客形态成功转型为具有“数字工坊”隐喻的学习系统界面。
2. 架构具备高扩展性：依托 `@xyflow/react` 现成生态模块，后续加新功能（如拖拽建图）门槛极低。
3. Tailwind 结合 React 的玻璃拟态设计极大提升了科技感，不流于俗套。

## 8. 本轮缺点与代价

1. Astro 集成 React 会稍微增加打包体积和渲染心智（客户端水合）。
2. 目前的节点坐标还是硬编码 `x,y` 或者部分随机，缺乏极度优美的自动布局（Layout Engine如 Dagre）支持。

## 9. 验证与结果

1. 构建运行 `npm run build` 成功。
2. 数据状态完全可控：锁定节点置灰，进行中的主线带呼吸灯效应，已完成节点连出的线条变为亮蓝色。

## 10. 本轮结论

使用 React Flow 是 Astro 制作高度交互“游戏化地图”的最优方案之一，既隔离了 DOM 复杂度，又保留了用 Tailwind CSS 自定义 UI 的灵活性。

## 11. 下一轮建议

1. **引入自动布局算法**：对庞大的树状图引入 `dagrejs` 进行自动化坐标排版。
2. **结合实际 Markdown**：点击“前往任务面板”时应当读取 `src/content` 中的相关笔记，展示实际交付件。
