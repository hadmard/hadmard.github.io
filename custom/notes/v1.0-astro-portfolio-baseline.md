# 版本记录：v1.0-astro-portfolio-baseline

## 1. 本轮目标

本轮目标是从空仓库建立一个可直接用于 `GitHub Pages` 的个人博客基线，并满足以下要求：

- 覆盖个人简历、项目经历、投资记录、个人思考四类内容
- 支持中英文双语切换
- 视觉风格现代、有个性，但不过度花哨
- 支持评论能力

这一轮属于：功能补充 + 结构基线建立 + 可维护性提升。

## 2. 背景与上下文

本仓库开始时几乎为空，没有现成业务代码，也没有已有版本记录。

因此本轮被定义为首轮基线建设。所有结构、技术选型、部署方式、内容组织和记录规范，都需要在这一轮完成首次落地。

另外，用户明确要求：

- 采用尽量新的、强体验、适合个人博客的技术栈
- 与 `hadmard.github.io` 绑定
- 最好带评论功能

## 3. 问题分析

表面问题是“做一个炫酷个人博客”，但真正的核心矛盾有三个：

1. 既要有足够强的视觉表达，又不能为了炫技牺牲 GitHub Pages 的静态部署稳定性。
2. 既要支持双语和后续长期写作，又不能把站点做成重型 CMS 或复杂前后端系统。
3. 评论功能要尽量轻量，否则会引入数据库、鉴权、后端托管等超出 GitHub Pages 边界的复杂度。

主要原因不是页面怎么画，而是要在“表达力、维护成本、部署边界、长期扩展性”之间找到平衡。

## 4. 候选方案比较

### 方案 A：Next.js / React 全栈博客

核心思路：
用常见 React 全栈框架做动态博客，再部署到支持 SSR 的平台。

优点：

- 生态非常丰富
- 交互能力强
- 容易接评论、后台和数据库

缺点：

- 与 GitHub Pages 的静态托管边界不匹配
- 工程复杂度明显更高
- 对“个人站点长期低成本维护”不够友好

放弃原因：

用户当前目标明确指向 GitHub Pages，本方案复杂度和部署模型都偏重。

### 方案 B：纯 HTML/CSS/JS 手写静态页

核心思路：
完全手写，避免框架依赖。

优点：

- 最轻
- 部署最简单
- 没有框架升级成本

缺点：

- 双语与文章内容扩展成本高
- 内容结构和文章详情页管理不友好
- 后续写作、文章路由、内容集合会变得凌乱

放弃原因：

适合极小页面，不适合需要持续扩写的个人内容站。

### 方案 C：Astro 6 + Tailwind CSS 4 + MDX + Astro i18n

核心思路：
以现代静态站框架承接内容系统、双语路由、MDX 文章与强视觉首页，同时保持 GitHub Pages 友好。

优点：

- 静态优先，性能与部署边界都合适
- 天然适合内容站与文章集合
- 双语、内容集合、静态构建体验成熟
- 能做出比传统模板更有叙事感的页面

缺点：

- 需要处理 Astro 6 的新内容集合写法
- 评论能力如果不接后端，需要依赖 GitHub 生态组件

最终选择原因：

它是在“最新、够强、够顺滑、适合 GitHub Pages”这几个维度上最平衡的方案。

## 5. 最终决策

最终采用：

- `Astro 6`
- `TypeScript`
- `Tailwind CSS 4`
- `MDX`
- `Astro i18n`
- `GitHub Pages Actions` 自动部署
- `utterances` 评论组件

关键设计决策如下：

1. 站点结构采用“单个主叙事首页 + 文章详情页”模式，而不是把四个模块拆成很多传统二级页面。
2. 中英文内容统一通过 `src/data/site.ts` 与 `src/content/thoughts/` 管理，避免散落在多个模板中。
3. 评论功能优先选用 `utterances`，原因是它只依赖 GitHub Issues，能够在已知仓库名的前提下直接接入；`Giscus` 功能更强，但需要额外的 repo/category id 配置，当前缺少必要仓库配置上下文。
4. `astro.config.mjs` 默认直接绑定到 `https://hadmard.github.io`。

## 6. 具体实现

本轮主要改动文件如下：

- `astro.config.mjs`
  绑定站点地址、开启国际化、接入 MDX / sitemap / Tailwind。
- `src/layouts/Layout.astro`
  建立统一 SEO、背景层和页面框架。
- `src/styles/global.css`
  建立整站视觉变量、共享卡片样式和文章排版。
- `src/data/site.ts`
  集中维护中英文首页内容。
- `src/content.config.ts`
  建立文章内容集合定义。
- `src/components/HomePage.astro`
  实现四大模块的首页叙事结构。
- `src/components/ThoughtPostPage.astro`
  实现文章详情页与侧边信息区。
- `src/components/Comments.astro`
  接入 `utterances` 评论。
- `src/components/LocaleSwitch.astro`
  实现中英文切换按钮。
- `src/pages/index.astro`
  中文首页入口。
- `src/pages/en/index.astro`
  英文首页入口。
- `src/pages/thinking/[slug].astro`
  中文文章详情路由。
- `src/pages/en/thinking/[slug].astro`
  英文文章详情路由。
- `src/content/thoughts/...`
  新建中英文示例文章内容。
- `.github/workflows/deploy.yml`
  GitHub Pages 自动部署工作流。
- `README.md`
  改为当前项目说明。
- `AGENTS.md`
  补齐仓库内的代理操作基线，并同步最新一期路径。

## 7. 本轮优点

当前方案相比空仓库或传统单页模板的主要提升：

- 建立了完整可运行的站点基线
- 双语结构已经打通
- 内容与展示层分离，后续扩写成本低
- 视觉上明显区别于常见“简历模板风”
- 文章详情页与评论能力都可直接继续扩展
- GitHub Pages 部署链路已经准备好

改善最明显的质量属性包括：

- 可维护性
- 可扩展性
- 内容复用性
- 静态部署稳定性
- 长期写作承接能力

## 8. 本轮缺点与代价

这一轮也保留了几个明确的 trade-off：

1. 个人真实信息尚未提供，因此首页中的履历、项目与投资内容是高质量占位文本，而非用户真实资料。
2. 评论功能当前使用 `utterances` 而不是 `Giscus`。
   这样做的好处是能立即接入，代价是交互形态和讨论体验不如 GitHub Discussions 方案丰富。
3. 当前首页采用强叙事的一页式布局，视觉更完整，但如果未来内容密度极高，可能仍需要拆成更细的专题页面。
4. 内容集合的类型提示虽然已经可用，但 Astro 6 仍会提示与 `z` 相关的 hints；这不影响构建和运行，但说明后续还可以继续向更“新”的内容 schema 写法靠拢。

## 9. 验证与结果

本轮实际做了以下验证：

- 运行 `npm run build`
  结果：通过。
- 运行 `npm run check`
  结果：通过，无错误。

验证确认了以下结论：

- 双语首页可以参与静态构建
- 文章详情页可以参与静态构建
- GitHub Pages 所需的产物输出正常
- 评论组件代码不会阻塞构建

暂未完成的验证：

- 未在真实 GitHub 仓库页面上完成一次完整发布验证
- 未在真实浏览器里完成评论授权与提交验证

原因：

- 当前工作在本地仓库中进行，未直接连接到线上仓库设置页面

剩余风险：

- 若 `hadmard/hadmard.github.io` 还未安装 `utterances` App，则评论框会显示但无法真正提交评论
- 若用户后续改为项目页仓库而非用户页仓库，需要重新确认 `site/base` 策略

## 10. 本轮结论

本轮最重要的结论是：

对于“GitHub Pages + 双语 + 个人内容站 + 炫酷但克制”的目标，`Astro 6 + Tailwind CSS 4 + MDX` 是非常合适的首轮基线。

被证实的假设：

- 强叙事首页可以和静态内容体系共存
- 评论功能不必依赖后端，也能先建立基线
- 双语内容集中管理会显著降低后续维护成本

本轮形成的新默认经验：

- GitHub Pages 优先采用静态优先内容架构
- 内容最好集中在 `src/data` 与 `src/content`，不要散在页面里
- 评论优先可用性，再逐步升级体验

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 用真实的履历、项目、投资记录替换 `src/data/site.ts` 中的占位内容。
   这是当前收益最高的下一步。
2. 给 `hadmard/hadmard.github.io` 安装 `utterances`，完成评论的真实启用。
   这是当前风险最低、最容易闭环的一步。
3. 如果你更偏好 GitHub Discussions 体验，再把评论层从 `utterances` 升级为 `Giscus`。
   这需要额外配置 Discussions 与 category id，建议放在真实内容替换之后再做。
