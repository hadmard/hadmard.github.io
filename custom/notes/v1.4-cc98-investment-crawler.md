# 版本记录：v1.4-cc98-investment-crawler

## 1. 本轮目标

本轮目标是为用户提供一个独立的 CC98 主题楼层爬虫，用于抓取指定主题的楼层消息，并把结果整理为本地投资记录。

这一轮属于：功能补充 + 工具基线建立。

优先级高的原因是，用户已经明确给出目标链接，并要求在仓库根目录单独起一个文件夹完成抓取与本地保存，这属于一项独立交付物，而不是站点内的小修小补。

## 3. 问题分析

表面上看，这次需求像是“写一个抓页面 HTML 的爬虫”。  
但实际验证后，核心矛盾并不是“怎么解析页面 DOM”，而是“目标内容并不在首屏 HTML 里，而是走鉴权 API”。

本轮确认到的关键事实有：

1. `https://www.cc98.org/topic/6450962` 首屏返回的是前端壳页面，不包含楼层正文。
2. 站点真实 API 域名来自前端配置文件：
   - `https://api.cc98.org`
3. 前端主题页楼层接口走的是：
   - `/Topic/{topicId}/post?from={offset}&size={limit}`
4. 目标主题当前在无授权情况下返回：
   - `401 Unauthorized`
   - `cannot_entry_board`

因此，本轮真正的问题不是“能不能写爬虫”，而是：

- 爬虫必须支持授权
- 没有用户自己的 CC98 token，就无法真正抓到这个主题内容

这也是本轮最需要沉淀下来的约束，否则后续代理很容易重复花时间去抓静态 HTML。

## 4. 候选方案比较

### 方案 A：直接抓网页 HTML 并解析 DOM

核心思路：

- 请求主题页面
- 解析页面里的楼层节点

优点：

- 从用户视角最直观
- 不需要理解前端内部接口

缺点：

- 实测首屏 HTML 不包含楼层正文
- 页面内容由前端脚本异步加载
- 即使抓到页面，也只是应用外壳

放弃原因：

不是正确的数据入口，继续坚持只会做出脆弱且无效的实现。

### 方案 B：用浏览器自动化强行等页面渲染后再抓

核心思路：

- 打开真实浏览器
- 等待页面前端把数据请求回来
- 再从已渲染 DOM 中提取楼层内容

优点：

- 理论上可复现用户浏览器里的最终显示结果
- 对复杂渲染页面适配性更强

缺点：

- 当前核心阻碍仍然是鉴权，不是渲染
- 没有用户登录态，浏览器自动化一样拿不到内容
- 依赖 DOM 结构，后续维护成本更高

放弃原因：

它解决的是“渲染层”，而不是当前真正的“授权层”问题。

### 方案 C：直接对接 CC98 API，并把抓取层与整理层拆开

核心思路：

- 通过 API 拉主题与楼层
- 原始 JSON 单独保存
- 再导出投资记录 Markdown/CSV

优点：

- 数据来源稳定
- 结果更容易复核
- 可以明确暴露鉴权需求
- 后续要做更强的结构化提取时也更容易扩展

缺点：

- 需要用户提供自己的 token
- 在无授权时只能先完成工具基线，不能完成真实抓取

最终选择原因：

这是当前最诚实、最稳健、最可继承的方案。

## 5. 最终决策

本轮最终采用方案 C，具体决策如下：

1. 在 `custom/` 下单独新增 `custom/cc98-investment-crawler/`
2. 使用 Node 22 原生 `fetch` 实现抓取，不新增第三方依赖
3. 脚本同时输出：
   - 原始 JSON
   - 投资记录 Markdown
   - 投资记录 CSV
4. 无 token 时明确报错
5. 即使有脚本，也不伪装成“已经完成实际抓取”，而是明确说明当前主题需要授权

换句话说，本轮的交付重点是：

- 建立可跑通的抓取工具骨架
- 明确当前目标主题的授权边界
- 把本地记录格式先落稳

## 7. 本轮优点

本轮方案相比“直接写死一个一次性抓取脚本”，主要优点有：

1. 数据抓取与结果整理分离，后续更容易继续做结构化提炼。
2. 原始 JSON 与整理结果同时保留，方便核对和追溯。
3. 不改仓库现有站点结构，风险集中在独立目录内。
4. 通过最小实现就把关键约束暴露出来，避免后续代理重复走错方向。

质量属性改善主要体现在：

- 可继承性
- 可复核性
- 可维护性
- 工程边界清晰度

## 8. 本轮缺点与代价

这一轮也有明显代价，必须明确写出：

1. 由于当前主题需要授权，本轮无法在没有用户 token 的前提下完成真实内容抓取。
2. 正文清洗目前只做轻量 UBB/HTML 去壳，还没有做更激进的投资要点结构化抽取。
3. 这轮输出更偏“保真记录”，还不是最终意义上的高质量投资研究笔记。
4. 如果 CC98 后续调整接口路径或字段名，脚本仍需要再跟一次接口变化。

也就是说，这一轮是主动接受“先把抓取基线和边界立住，再继续精修记录质量”的取舍。

## 9. 验证与结果

本轮完成的验证：

- 读取 `https://www.cc98.org/static/config.json`
- 确认 API 域名为 `https://api.cc98.org`
- 验证目标主题无授权访问结果为：
  - `401 Unauthorized`
  - `cannot_entry_board`
- 本地最小运行验证计划：
  - `node custom/cc98-investment-crawler/crawl-cc98-topic.mjs --help`
  - `node custom/cc98-investment-crawler/crawl-cc98-topic.mjs --topic https://www.cc98.org/topic/6450962`

当前确认结论：

- 脚本路径和输出结构已经落地
- 当前最大阻碍不是代码，而是目标主题访问授权

未完成验证：

- 未能完成真实主题抓取

原因：

- 当前没有用户自己的 CC98 token

剩余风险：

- 真正拿到授权后，可能还需要根据实际返回字段做一次小修

## 10. 本轮结论

本轮最重要的结论是：

这个任务的正确切入点不是“抓网页”，而是“做一个支持授权的 CC98 API 抓取器”。

被证实的判断：

- 主题页本身不包含可直接抓取的楼层 HTML
- API 域名和楼层接口可以从前端包中定位出来
- 当前目标主题确实需要授权

被否定的假设：

- “只要请求主题页面 HTML 就能提取楼层”这一假设不成立

后续应默认继承的经验：

- 遇到类似 CC98 主题抓取需求，先确认前端配置里的 `apiUrl`
- 先确认授权边界，再决定是 DOM 抓取还是 API 抓取

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 让用户提供一个可用的 CC98 token，这是当前收益最高的动作。
2. 在拿到真实楼层数据后，再补一层“投资观点/标的/时间线”提取，这是最直接提升记录质量的动作。
3. 如果后续会长期抓 CC98 内容，可以再补一个更细的配置层，例如多个主题批量抓取与输出命名规则。
