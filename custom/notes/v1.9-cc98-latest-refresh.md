# 版本记录：v1.9-cc98-latest-refresh

## 1. 本轮目标

本轮目标是刷新 CC98 主题 `6450962` 的最新楼层内容，并把更新后的本地数据重新同步到当前站点构建产物。

这一轮属于功能补充中的增量同步。优先级高的原因很直接：上一轮沉淀下来的抓取链路已经可用，本轮只需要把新增楼层及时落到本地文件和首页展示，不需要再改数据结构和展示形态。

## 2. 背景与上下文

本轮承接：

- `custom/notes/v1.8-learning-ui-redesign.md`
- `custom/notes/v1.7-cc98-pages-sync.md`
- `custom/notes/v1.6-cc98-authenticated-crawl.md`

其中与本次任务直接相关的是 `v1.7` 和 `v1.6`：

1. `v1.6` 已经确认 CC98 可以通过 OpenID 密码登录换取 token，并稳定抓取授权主题。
2. `v1.7` 已经把本地抓取结果接进首页构建流程，首页直接读取 `custom/cc98-investment-crawler/output/topic-6450962/` 下的 JSON。

`v1.8` 主要是前端 UI 重构，与 CC98 抓取本身不直接相关，但它意味着当前工作区存在一批未提交的前端变更，本轮需要避免误改这些内容。

## 3. 问题分析

当前问题不是“怎么抓”，而是“怎么在不打扰现有前端改动的前提下完成最新同步”。

核心约束有三个：

1. 抓取目标仍然是同一个主题，属于增量刷新，不应重构爬虫。
2. 当前工作区已经存在用户自己的前端改动，不能把这轮更新扩散到无关源码。
3. 首页展示逻辑已经约定为原文直出，因此不能借本轮同步擅自改成结构化交易视图。

## 4. 候选方案比较

### 方案 A：只更新本地 JSON，不重建站点

核心思路：

- 运行爬虫刷新 `output/topic-6450962`
- 不执行 `npm run build`

优点：

- 改动最小
- 不会触发静态产物变化

缺点：

- 本地站点展示仍然停留在旧楼层数
- 无法满足“然后更新”的完整语义

放弃原因：

这只能算数据刷新，不算展示同步完成。

### 方案 B：刷新数据后按当前工作区状态重建站点

核心思路：

- 运行现有爬虫抓取最新内容
- 保持首页逻辑不变
- 直接重建静态产物，让页面消费最新 JSON

优点：

- 延续现有链路，不引入新逻辑
- 能完整闭环到页面层

缺点：

- 构建产物会同时反映当前工作区已有的前端改动

最终选择原因：

这是唯一能在不改展示逻辑的前提下完成“最新抓取 + 页面更新”的方案。需要接受的边界是：本轮构建基于当前工作区状态，而不是回退用户已有前端改动。

## 5. 最终决策

本轮最终采用方案 B，具体做法如下：

1. 继续使用现有 `crawl-cc98-topic.mjs` 抓取同一主题。
2. 沿用用户名密码自动登录链路，不修改授权方式。
3. 不改首页数据结构和展示文案，只刷新底层数据文件。
4. 在当前工作区状态下执行构建，让首页显示新增楼层。
